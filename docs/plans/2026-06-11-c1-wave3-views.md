# C1 Wave 3 — views: findings & the grid-views dead-code cleanup

Wave 3 of the C1 Product Grid Backbone→React migration was scoped (from the Wave 0
audit) as "views": pagination + grid-view dropdown + no-data. Investigating the
sources before implementation overturned that scope.

## Findings

- **`grid-views/view.js` is dead code.** The audit listed it as the grid-view
  dropdown to migrate, but it has **zero references** anywhere in `src`, `tests`,
  PHP, YAML or templates (only its own requirejs alias). Nothing instantiates it; its
  historical mount in `datagrid-builder` was already a no-op (audit danger note). Its
  two collaborators — `grid-views/collection.js` and `grid-views/model.js` — are used
  **only** by `view.js` (and each other), so the whole `grid-views/` directory is a
  dead island.
- **The real product-grid view selector is `pim/grid/view-selector`**
  (`form_extensions/product/index.yml:95` → `pimui/js/product/grid/view-selector`). It
  is a substantial subsystem of ~8 modules (`view-selector`, `selector`,
  `secondary-actions`, `line`, `create-view`, `save-view`, `remove-view`, `current`)
  built on select2 + `FormBuilder` view CRUD + analytics. A React migration of it is a
  dedicated wave of its own, not a small "dropdown" change.
- **Pagination** (`pagination.js` + `pagination-input.js`) is shared grid chrome used
  by ~8 grids app-wide, with rescore-window/mediator/DOM-placement logic — also a
  dedicated cross-grid effort, not part of this cleanup.
- **no-data** (`grid.js` `_updateNoDataBlock`) stays with Wave 5 (core/state).

## This change

Delete the dead `grid-views/` island and its requirejs aliases:

- `Resources/public/js/datagrid/grid-views/view.js`
- `Resources/public/js/datagrid/grid-views/collection.js`
- `Resources/public/js/datagrid/grid-views/model.js`
- the three `oro/datagrid/grid-views/{view,model,collection}` aliases in
  `PimDataGridBundle/Resources/config/requirejs.yml`.

No behaviour changes — the code was unreachable.

## Next

The genuine remaining C1 surfaces are all large/shared/high-risk: the
`pim/grid/view-selector` subsystem, the shared pagination, the filters (Wave 4), and
the core/state engine (Wave 5, which also owns no-data and the select-all header).
Each warrants its own brainstorm → spec → plan cycle.
