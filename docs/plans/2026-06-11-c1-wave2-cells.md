# C1 Wave 2 ‚Äî Product grid read-only cells ‚Üí React

Wave 2 of the C1 Product Grid Backbone‚ÜíReact migration (Strangler Fig). Grounded in
the Wave 0 audit (`docs/plans/2026-06-10-product-grid-audit.md`, ¬ß4 wave 2 + ¬ß5 bridge).

## Scope

Read-only **data** cells of the product grid only. The select-all **header**
checkbox is deferred to wave 5 (it is injected into Backgrid's `thead` by
`column-form-listener.js:22-29` and feeds the selection pipeline), and
`select-row-cell` (selection) likewise waits for the wave-5 state engine.

| --------------------------------- | --------------------------------------------------- |
| `enabled`                         | `oro/datagrid/enabled-cell`                         |
| `product-and-product-model-label` | `oro/datagrid/product-and-product-model-label-cell` |

## Decisions

- **Per-type bridge, not a generic `reactCell`.** The audit's verdict: do NOT
  generalize `reactCell.tsx` (dead code for the product grid; leaks because it
  never unmounts; recreates providers each render). Instead, a shared
  `ReactCellBase` with a fixed lifecycle, which each `{{type}}-cell` module
  extends ‚Äî keeping the `oro/datagrid/{{type}}-cell` module name AND the YAML
  `frontend_type` unchanged (Strangler Fig, like wave 1's display selector).
- **ES class subclasses.** Backbone's `.extend` cannot subclass an ES class
  (`Parent.apply(this)` throws on a class constructor), so each cell is an ES
  `class X extends ReactCellBase` in a `.tsx` file. Module resolution: the
  requirejs alias gains the `.tsx` extension (mirroring `delete-confirm.ts` and
  the DQI `quality-score-badge-cell.tsx`).
- **Incremental rollout.** PR 1 lands the base + one exemplar cell, fully tested,
  to prove the pattern at low risk. Follow-up PR(s) replicate it for the other 4.

## The `ReactCellBase` lifecycle fix

`render()` wraps `this.reactContent()` in `ThemeProvider` + `DependenciesProvider`
and `ReactDOM.render`s into the cell `el`. `remove()` calls
`ReactDOM.unmountComponentAtNode(this.el)` **before** `super.remove()` detaches the
node ‚Äî fixing both the detached-tree leak Backgrid would otherwise cause on every
collection refresh and React 18's "unmount from a detached root" warning. Legacy
`ReactDOM.render` (not `createRoot`) is kept per the audit's Selenium constraint
(`base.ts:263-272`).

Subclasses stay thin: override `reactContent()` to derive props from `this.model`
and return a presentational component, or `null` for an empty cell.

## PR 1 ‚Äî base + `enabled` exemplar

- `react-cell-base.tsx` ‚Äî the shared base (above).
- `EnabledBadge.tsx` ‚Äî presentational badge; reproduces the legacy
  `enabled-cell.html` markup (`AknBadge--{enabled,disabled}` / `status-{...}`),
  which is the toggle-status Behat/CSS contract.
- `enabled-cell.tsx` ‚Äî `class EnabledCell extends ReactCellBase`; keeps the
  `product_model` blank-cell guard (PIM-6493) and derives `enabled`/`label`.
- requirejs alias updated to `enabled-cell.tsx`; old `.js` removed.

### Tests

- `EnabledBadge.unit.tsx` ‚Äî both status states (classes + label).
- `react-cell-base.unit.tsx` ‚Äî render mounts content; **remove unmounts** (kills
  the leak-fix mutant) and chains to Backgrid remove; base renders an empty cell.
- `enabled-cell.unit.tsx` ‚Äî `reactContent()`: product/product-model guard,
  enabled vs disabled derivation, label key.
- Mutation: the 3 `.tsx` sources are added to the Stryker per-PR scope
  (`stryker.jest.js` testMatch).
- Behat/Playwright: the existing toggle-status / status-display features assert
  on the preserved `AknBadge`/`status-*` markup ‚Äî no new feature needed.

## PR 2 ‚Äî completeness + complete-variant-product

Reading the four remaining cells refined the scope:

- **`completeness` + `complete-variant-product`** ‚Äî migrated here. Both are
  StringCell-based badge cells with genuine conditional logic (colour-coded
  `AknBadge` by ratio, "not available" for the wrong document type, dash for
  missing data), a clean `ReactCellBase` fit. They share a `CompletenessBadge`
  presentational component. Tests cover `reactContent()` per branch + the badge.
- **`label`** ‚Äî left as Backgrid. It renders plain text plus a `title` attribute
  and a dynamic `<td>` class; wrapping a text node in a per-cell `ReactDOM.render`
  (√ó25 rows) adds lifecycle overhead with no UI benefit. Not a React candidate.
- **`image`** ‚Äî initially deferred, **migrated in PR 3**. Although it extended
  `ImageCell`, that base itself extends `StringCell`, so the cell fits
  `ReactCellBase` after all: its ~3 lines of ImageCell logic (resolve the
  thumbnail + placeholder urls via `MediaUrlGenerator`, swap to the placeholder
  on a load error) are re-implemented in a `ProductImage` presentational
  component. The base `image-cell.js` (used by the association grid) stays
  Backgrid; only the product-grid `product-and-product-model-image-cell` moves.

## PR 3 ‚Äî image cell

`product-and-product-model-image-cell` ‚Üí `ReactCellBase`, rendering a shared
`ProductImage` component. Preserves the `AknGrid-image[--withLayer]` markup, the
product-model stacked layer, and the one-shot error‚Üíplaceholder fallback (the
legacy `.one('error', ‚Ä¶)`). Tests cover the plain/stacked variants, the fallback
swap, and `reactContent()` url derivation. This **closes the Wave 2 cells**:
enabled, completeness, complete-variant and image are React; label stays Backgrid.
